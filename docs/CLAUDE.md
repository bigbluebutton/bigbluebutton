# Agent notes for working in `docs/`

This is the Docusaurus site for docs.bigbluebutton.org. Read this before
spinning up local builds — there are a few non-obvious things.

## Pick the right branch

- **`develop`** is the deploy branch. `.github/workflows/deploy-docs.yml`
  is the only path to docs.bigbluebutton.org. Its `build.sh` orchestrates
  pulling per-version content from `v3.0.x-develop` etc. into a
  multi-version site.
- **`v3.0.x-develop`** (and other `vX.Y.x-*` branches) hold the *content*
  for that version. The single-version config here boots
  `npx docusaurus start` directly with no `build.sh` dance. Prefer this
  branch for content-only edits to the 3.0 docs — just remember it is
  not deployed on its own; the change rides along on the next `develop`
  deploy.

## Local build/serve gotchas

- **Run `npm ci` first.** `node_modules` drifts from the lockfile across
  branch switches and stale checkouts. Don't trust the installed plugin
  versions until you've synced.
- **`docusaurus serve` needs `NODE_ENV=production`** to match the
  built site's baseUrl. Without it, serve uses the dev baseUrl `/docs/`,
  every asset 404s, and you'll see `Unexpected token '<'` errors that
  look like a real bug but are just an HTML 404 page being parsed as JS.

  ```bash
  NODE_ENV=production npx docusaurus serve --host 127.0.0.1 --port 3010 --no-open
  ```

- **On `develop`, `npx docusaurus start` fails without `versioned_docs/`.**
  The clean fix is `./build.sh --remote <upstream-remote>`, but that
  switches branches — destructive if there are uncommitted changes. For
  quick local-only validation, temporarily set
  `includeCurrentVersion: true, versions: {}` in `docusaurus.config.js`
  and remove redirects targeting versioned paths (`/2.6/...`, `/2.7/...`).
  Restore both before committing.
- **`onBrokenLinks: 'throw'`** is enabled. The build log prints the
  valid path list when it fails — use that as the authoritative source.
- The search index (`search-index.json`) is only generated by
  `docusaurus build`, never by `docusaurus start`. In dev mode the
  `/search` page shows "search index is only available when you run
  docusaurus build" — expected, not a regression.
- Clean `docs/build/` and `docs/.docusaurus/` between runs when you
  switch branches. Both are gitignored.

## Validating JS-driven features

Playwright is already installed at
`bigbluebutton-tests/playwright/node_modules/playwright` — no extra
install needed. Use it for anything you can't see in the rendered HTML
(search dropdown, version selector, sidebar collapse, console errors).

```js
const { chromium } = require('/abs/path/to/bigbluebutton-tests/playwright/node_modules/playwright');
const browser = await chromium.launch();
const page = await (await browser.newContext()).newPage();
page.on('pageerror', e => console.log('ERR:', e.message));
await page.goto('http://127.0.0.1:3010/');
// ...interact, assert
await browser.close();
```

## Source of truth

When updating tables/values that document server behaviour (e.g. the
"Other meeting configs available" table in
`administration/customize.md`), cross-reference:

- `bigbluebutton-web/grails-app/conf/bigbluebutton.properties` —
  authoritative for bbb-web defaults
- `docs/data/create.tsx` — authoritative for `/create` API parameters
  and which property defaults are overwritable per meeting

A bbb-web property is "overwritable" if there is a matching entry in
`create.tsx`. The create-side name sometimes differs (e.g.
`defaultMeetingLayout` → `meetingLayout`, `defaultMaxUsers` →
`maxParticipants`, `defaultKeepEvents` → `meetingKeepEvents`). The
`docs/administration/customize.md` table calls out these renames
inline.
