- When writing unit tests for the bbb-api the testing functions should be implemented in a Go table testing style. Only use Go standard library features. Testing/assertion libraries and frameworks should never be used. The goal should be to achieve 100% code coverage on all code paths being tested by the function. Unit tests should be placed in a new file in the same directory as the function being tested. The new file should have the same name as the file where the function resides with "_test" appended to the name before the ".go" extension. There should only be as many test cases as necessary to test every code path and ensure the logic of the function is accurate. Avoid adding superfluous test cases. Tests for full pipeline flows, typically found in "flow_test.go" files, should not use table testing. Each test case should be placed in its own test function to allow them to more easily map on to integration test scenarios. Test cases should cover scenarios such as successful flow execution, failure of any external service calls, and flow execution error. It is not necessary to create test cases for every kind of response or error that may be returned by a flow since tests for the individual steps in the flow will cover all those cases. Flows that branch or merge conditionally will have additional test cases covering such scenarios. It is also not necessary to do in-depth validation on the output of the flow. If flow execution is successful a message with a non-nil payload and nil error should be returned. If flow execution is unsuccessful an error and a message, typically with a nil payload, should be returned. Performing nil checks on the ouput message payload and error should typically be sufficient for most flow execution test cases.
